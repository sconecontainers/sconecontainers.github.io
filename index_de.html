SCONE in Kürze

Übersicht über SCONE's Alleinstellungsmerkmale

  • Transparente

    Verschlüsselung

    —

    SCONE verschlüsselt Daten und Netzwerkverkehr transparent. Auf diese Weise werden Daten vor unauthorisiertem Zugriff von Seiten des Betriebssystems, des Hypervisors oder jeglicher Software geschützt.

    Mehr dazu

  • Transparente

    Attestierung von Programmen

    —

    Programme werden codeunabhängig von SCONE attestiert, um sicherzustellen, dass nur fehlerfreie, unmodifizierte Programme in der SGX-Enklave ausgeführt werden. So wird auch verhindert, dass Programme von Malware infiziert werden.

  • Curated

    Images

    —

    SCONE unterstützt eine Reihe von Curated Images und vereinfacht so die Anwendungsentwicklung. Für die meisten Standardservices bieten wir ein SCONE Image an, das innerhalb einer SGX-Enklave ausgeführt wird. Das Image kann unter Nutzung Docker Compose- oder Stack-Files angepasst werden. Beispiele sind in unserem ausführlichen SCONE Tutorial zu finden.

  • Sichere

    Compose-Files

    —

    SCONE unterstützt sichere Compose-Files, um Geheimnisse zu schützen, die in Docker Compose, Stack-Files, als auch Kubernetes Metadaten-Files sichtbar sind.

    Mehr dazu

  • Einfache Nutzung

    von sicheren Enklaven

    —

    SCONE führt Programme innerhalb sicherer Enklaven aus - ohne Source-Code-Änderungen. Wir empfehlen die Anwendungen zu rekompilieren, unterstützen aber die Ausführung nativer Apline Anwendungen innerhalb von Containern. So werden auch Angreifer mit Root-Access daran gehindert, Geheimnisse von diesen Programmen zu stehlen.

    Mehr dazu

  • Automatische Bereitstellung

    von Schlüsseln

    —

    SCONE hilft bei der Konfiguration von Programmen mit Geheimnissen, die weder gelesen noch von Angreifern modifiziert werden können - sogar wenn sie bereits Kontrolle über das Betriebssystem und/oder den Hypervisor gewonnen hätten. Dies geschieht codeunabhängig von der Anwendung ohne Änderungen des Source-Codes.

  • Anwendungsbezogene

    Sicherheit

    —

    Anwendungssicherheit, selbst in nicht vertrauenswürdigen Umgebungen.

    Mehr dazu

  • Cross-Compiler

    Support

    —

    SCONE unterstützt Cross-Compiler für C, C++, Rust, Go und einen Interpreter für Python.

    Mehr dazu

NEARDATA in the confidential computation realm

One of the aspects of the efforts is to bring confidentiality to the computation performed by the systems.
SCONE has been selected as the key framework to provide confidential computation to Flower ML, Pravega, and Lithops.

USE CASE: SURGERY

Federated Learning is a method to employ machine learning over distributed sources, by moving the computation to the data (check here for more information).

Pravega is a distributed storage system based on streaming, used during surgical procedures to capture and preprocess laparoscopic video streams before storing them. These captured videos serve as the primary data source for machine learning model training.

Due to strict privacy regulations in healthcare, direct data sharing between hospitals is typically prohibited. To address this, Federated Learning is applied using the Flower ML framework, which enables decentralized model training by exchanging only model updates rather than raw data.

SCONE provides an end-to-end trusted execution environment for both Pravega and Flower, ensuring data confidentiality from intraoperative capture to distributed model training.

USE CASE: METABOLOMICS

The spatial metabolomics annotation is used by the areas of biology, medicine, and pharmacology. It computes gigabytes of data in order to detect metabolite structures and other molecules.

Lithops is a multi-cloud serverless computing framework, using Metaspace to perform the spatial metabolomics annotation in a decentralized manner, by submitting slices of computation jobs to clusters in cloud.

MinIO is an open source cloud object storage system compatible with S3 and used by Lithops to store data, as staging area, and as pipeline execution synchronization.

SCONE can support Lithops execution in cloud platforms with confidential containers for Kubernetes and Singularity compute backends, as well as for MinIO storage backend.

CONFIDENTIAL COMPUTING WITH SCONE

The SCONE platform is used to protect data (like PII and training data), and code (like Python programs) used by Lithops, Flower ML and Pravega client. SCONE covers data at rest (i.e., on disk), data in flight (i.e., on the network), and data in use (i.e., in the main memory), enforcing confidentiality and integrity of the data, the code and the computation.

In the surgonomics use case, SCONE provides the security for both Pravega client and Flower ML, enforcing confidentiality throughout the workflow, from the surgery room toward the processing the large amounts of data.

In the metabolomics use case, SCONE provides the security for Lithops, enforcing confidentiality throughout the workflow, from the data acquisition toward its processing in the cloud clusters.

Anwendungs-bezogene Sicherheit

SCONE bietet anwendungsorientierte Sicherheit. So können selbst in nicht vertrauenswürdigen Umgebungen Vertraulichkeit und die Integrität der Anwendung garantiert werden. In nicht vertrauenswürdigen Umgebungen muss ein Schutz gegen Angreifer gewährleistet sein, die Root-Access erlangt haben und auf diese Weise Anwendungsdaten lesen oder modifizieren können.

SCONE unterstützt selbstredend moderne Cloud-native Anwendungen, die sich aus mehreren Microservices zusammensetzen. Unsere generelle Empfehlung ist, jeden Microservice in einer Intel SGX Enklave laufen zu lassen. SCONE schützt eine Anwendung vor Angriffen über das Betriebssystem, anderen Anwendungen sowie internen APIs der Anwendung. In vielen Fällen müssen Microservices lediglich mit unseren SCONE Crosscompilern rekompiliert werden.

Intel SGX Enklaven

Intel SGX Enklaven werden bei SCONE zum Schutz von Anwendungskomponenten genutzt.

SCONE ermöglicht Entwicklern die Ausführung ihrer Anwendungen innerhalb sicherer SGX-Enklaven. SGX-Enklaven schützen Daten vor dem Zugriff jeglicher anderer Software - auch dem Betriebssystem. Die Anwendungsdaten sind im Besonderen auch gegen Angreifer mit Root-Access geschützt. Ein Root-User ist nicht in der Lage den Hauptspeicher einer Anwendung zu löschen, um Zugriff auf alle Schlüssel zu erhalten. In vielen Fällen sind Konfigurationsdateien lediglich durch das File-System geschützt. Ein Root-User könnte diese, sowie alle möglicherweise enthaltenen Geheimnisse lesen. Mithilfe von SCONE können diese Konfigurationsdateien verschlüsselt werden und sind somit geschützt.

SCONE Shielding

SCONE unterstützt transparente Verschlüsselung von Dateien, Netzwerk-Traffic und stdin/stdout.

Legacy-Services wie memcached wurden zur Ausführung in vertraulichen Umgebungen erstellt. Daher wird TLS bislang nicht unterstützt. SCONE bietet transparente Verschlüsselung von Netzwerk-Traffic; dazu kann der Netzwerk-Schutz als Teil des Stack/Compose-Files genutzt werden.

Während manche Anwendungen (wie MySQL) einige ihrer Daten verschlüsseln können, bieten die meisten Anwendungen keine Datenverschlüsselung an. Aus diesem Grund stellen wir in SCONE einen Mechanismus zur Verfügung, um Daten transparent zu verschlüsseln. Dieser Mechanismus wird über das Stack/Compose-File einer Anwendung aktiviert.

Weiterhin bieten wir einen Mechanismus, um stdin, stdout and stderr transparent zu verschlüsseln. Auch dieser wird über das Stack/Compose-File der Anwendung aktiviert.

Einige Anwendungen beziehen ihre Geheimnisse über Umgebungsvariablen oder Kommandozeilen-Argumente. SCONE gewährleistet mit dem SCONE Konfigurations- und Attestierungs-Service einen sicheren Transfer der Umgebungsvariablen und Kommandozeilenargumente.

Paper zu SCONE

Folgende wissenschaftliche Paper beschreiben einige technische Aspekte von SCONE. The following papers describe some of the technical aspects of SCONE.

SCONE: Secure Linux Containers with Intel SGX, USENIX, OSDI 2016

This paper describes how we support unmodified applications inside of enclaves. The focus is on our asynchronous system call interface.

Authors: Sergei Arnautov, Bohdan Trach, Franz Gregor, Thomas Knauth, André Martin, Christian Priebe, Joshua Lind, Divya Muthukumaran, Daniel O'Keeffe, Mark L Stillwell, David Goltzsche, Dave Eyers, Rüdiger Kapitza, Peter Pietzuch, Christof Fetzer

Media: pdf, slides audio

Abstract: In multi-tenant environments, Linux containers managed by Docker or Kubernetes have a lower resource footprint, faster startup times, and higher I/O performance compared to virtual machines (VMs) on hypervisors. Yet their weaker isolation guarantees, enforced through software kernel mechanisms, make it easier for attackers to compromise the confidentiality and integrity of application data within containers. We describe SCONE, a secure container mechanism for Docker that uses the SGX trusted execution support of Intel CPUs to protect container processes from outside attacks. The design of SCONE leads to (i) a small trusted computing base (TCB) and (ii) a low performance overhead: SCONE offers a secure C standard library interface that transparently encrypts/decrypts I/O data; to reduce the performance impact of thread synchronization and system calls within SGX enclaves, SCONE supports user-level threading and asynchronous system calls. Our evaluation shows that it protects unmodified applications with SGX, achieving 0.6x–1.2x of native throughput.

SGXBounds: Memory Safety for Shielded Execution, EuroSys 2017

To protect the code running inside of an enclave, we implemented a novel bounds checker for enclaves. While we had expected to just be able to use MPX, we had to realized that MPX does not perform that well inside of enclaves. For details regarding the overheads, please see this paper. This won the best paper award of EuroSys 2017.

Authors: D. Kuvaiskii, O. Oleksenko, S. Arnautov, B. Trach, P. Bhatotia, P. Felber, C. Fetzer

Media: pdf

Abstract: Shielded execution based on Intel SGX provides strong security guarantees for legacy applications running on untrusted platforms. However, memory safety attacks such as Heartbleed can render the confidentiality and integrity properties of shielded execution completely ineffective. To prevent these attacks, the state-of-the-art memory-safety approaches can be used in the context of shielded execution. In this work, we first showcase that two prominent software- and hardware-based defenses, AddressSanitizer and Intel MPX respectively, are impractical for shielded execution due to high performance and memory overheads. This motivated our design of SGXBounds -- an efficient memory-safety approach for shielded execution exploiting the architectural features of Intel SGX. Our design is based on a simple combination of tagged pointers and compact memory layout. We implemented SGXBounds based on the LLVM compiler framework targeting unmodified multithreaded applications. Our evaluation using Phoenix, PARSEC, and RIPE benchmark suites shows that SGXBounds has performance and memory overheads of 18% and 0.1% respectively, while providing security guarantees similar to AddressSanitizer and Intel MPX. We have obtained similar results with four real-world case studies: SQLite, Memcached, Apache, and Nginx.

FFQ: A Fast Single-Producer/Multiple-Consumer Concurrent FIFO Queue, IPDPS 2017

This paper describes our new lock-free queue for our asynchronous system calls.

Authors: Sergei Arnautov, Pascal Felber, Christof Fetzer and Bohdan Trach

Media: pdf

Abstract: With the spreading of multi-core architectures, operating systems and applications are becoming increasingly more concurrent and their scalability is often limited by the primitives used to synchronize the different hardware threads. In this paper, we address the problem of how to optimize the throughput of a system with multiple producer and consumer threads. Such applications typically synchronize their threads via multi- producer/multi-consumer FIFO queues, but existing solutions have poor scalability, as we could observe when designing a secure application framework that requires high-throughput communication between many concurrent threads. In our target system, however, the items enqueued by different producers do not necessarily need to be FIFO ordered. Hence, we propose a fast FIFO queue, FFQ, that aims at maximizing throughput by specializing the algorithm for single-producer/multiple-consumer settings: each producer has its own queue from which multiple consumers can concurrently dequeue. Furthermore, while we pro- vide a wait-free interface for producers, we limit ourselves to lock-free consumers to eliminate the need for helping. We also propose a multi-producer variant to show which synchronization operations we were able to remove by focusing on a single producer variant. Our evaluation analyses the performance using micro- benchmarks and compares our results with other state-of-the-art solutions: FFQ exhibits excellent performance and scalability.
